## 一、思考题 1：从 N-gram 到 Transformer
### 1
#### 计算 `agent works` 在 Bigram 模型下的概率：
句子 `agent works` 的概率约为 **0.167**

#### 马尔可夫假设含义,以及N-gram模型存在哪些根本性局限
马尔可夫假设：预测当前词时，不需要考虑前面所有的历史信息，只需要考虑前面有限数量的词
- Bigram 根据前面 1 个词预测下一个词
- Trigram 根据前面 2 个词预测下一个词
- N-gram 根据前面 N-1 个词预测下一个词
这个假设把复杂的语言概率问题转化成了可以通过语料库计数解决的问题，但也存在以下局限：
**数据稀疏**：如果一个词语组合没有在训练语料中出现，它的概率可能直接被计算为 0。平滑算法可以缓解，但不能从根本上解决这个问题
**不理解语义**：N-gram 把词当作彼此独立的符号。即使 `agent` 和 `robot` 含义相近，模型也无法把从 `agent learns` 学到的规律泛化到 `robot learns`
**上下文窗口固定**：模型只能查看前面有限数量的词，无法利用更早的长距离信息
**组合数量快速增长**：随着 N 和词表增大，需要统计、存储的词语组合会迅速增加，数据和存储成本都很高

#### 神经网络语言模型（RNN/LSTM）和Transformer分别是如何克服N-gram模型局限的？它们各自的优势是什么？
RNN/LSTM：
神经网络语言模型首先通过 Word Embedding（词嵌入），把离散的词转换成连续向量。语义相近的词在向量空间中也可能比较接近，因此模型获得了比 N-gram 更好的语义泛化能力。
RNN 又引入了 Hidden State（隐藏状态），将前面 Token 的信息不断传递到后面，因此不再受固定 N-gram 窗口的直接限制。LSTM 在 RNN 的基础上加入遗忘门、输入门和输出门，让模型能够更有选择地保留或丢弃信息，从而缓解普通 RNN 的梯度消失和长期依赖问题。

RNN/LSTM 的主要优势包括：

- 能够处理长度不固定的序列；
- 能通过词向量学习一定的语义关系；
- 能将历史信息传递到后续位置；
- 相比 N-gram，面对没有见过的相似表达时具有更好的泛化能力。

不过，RNN/LSTM 必须按照 Token 顺序串行计算，长距离信息仍可能逐渐丢失，因此不容易进行大规模并行训练。

#### Transformer

Transformer 使用 Self-Attention（自注意力），让每个 Token 可以直接计算上下文中其他 Token 与自己的相关程度，再汇总最相关的信息。两个距离很远的 Token 不再需要像 RNN 那样经过多个时间步逐步传递信息，因此更容易学习长距离依赖。

同时，Transformer 没有 RNN 的循环依赖。在训练阶段，一段文本中的多个位置可以并行计算，更适合利用 GPU 扩大模型、数据和训练规模。由于 Self-Attention 本身不知道 Token 的先后顺序，Transformer 还会加入 Positional Encoding（位置编码）。

Transformer 的主要优势包括：

- 更容易捕捉长距离上下文关系；
- 训练阶段可以并行计算；
- 更适合 GPU 和大规模预训练；
- 可以通过大规模数据获得更强的语义理解和泛化能力。

### 小结

这段发展过程可以概括为：

```text
N-gram：统计固定窗口中的词语组合
   ↓
RNN：通过隐藏状态传递历史信息
   ↓
LSTM：通过门控机制改善长期记忆
   ↓
Transformer：通过 Self-Attention 建立直接关系，并实现训练并行化
```

语言模型的发展，本质上是在持续解决两个问题：**如何更好地表示语言的语义，以及如何更有效地利用上下文信息。**

## 思考题 2：Transformer 架构

### 自注意力机制（Self-Attention）的核心思想是什么？

Self-Attention 的核心思想是：序列中的每个 Token 都会根据当前上下文，判断其他 Token 与自己的相关程度，再按照相关程度汇总信息，形成包含上下文的新表示。

例如，在句子 `The agent learns because it is intelligent.` 中，模型理解 `it` 时，应该重点关注前面的 `agent`。Self-Attention 会为 `agent` 分配较高的注意力权重。

在计算过程中，每个 Token 会生成三个向量：

- **Query（Q）**：当前 Token 想查询什么信息；
- **Key（K）**：每个 Token 可以被匹配的特征；
- **Value（V）**：每个 Token 实际携带的信息。

其核心过程可以简化为：

```text
Query 和 Key 计算相关程度
        ↓
得到 Attention 权重
        ↓
根据权重汇总 Value
        ↓
形成包含上下文的新 Token 表示
```

因此，同一个词在不同上下文中可以产生不同的表示。例如，“苹果很好吃”和“苹果发布了新手机”中的“苹果”，会根据上下文分别表示水果和公司。

### 为什么 Transformer 能并行处理序列，而 RNN 必须串行处理？

RNN 通过 Hidden State 传递历史信息：

```text
h1 → h2 → h3 → h4
```

计算第 `t` 个 Token 的隐藏状态时，需要先获得第 `t-1` 个 Token 的隐藏状态，因此必须按顺序串行计算，难以同时处理多个位置。

Transformer 没有 RNN 的循环结构。训练时，它可以把整段 Token 同时转换成 Q、K、V 矩阵，再通过矩阵运算同时计算多个 Token 之间的关系：

```text
Token 1 ┐
Token 2 ├→ 同时进行 Attention 计算
Token 3 ┤
Token 4 ┘
```

因此，Transformer 更适合使用 GPU 进行大规模并行训练，也更容易扩大模型参数、训练数据和计算规模。

需要注意：Transformer 在训练阶段可以并行处理多个位置，但 Decoder-Only 模型生成回答时，仍然需要逐 Token 生成。输出越长，通常生成时间也越长。

### 位置编码（Positional Encoding）起什么作用？

Self-Attention 可以计算 Token 之间的关系，但它本身不天然知道 Token 的先后顺序。例如，“我打你”和“你打我”包含相近的 Token，但顺序不同，意思也完全不同。

因此，Transformer 会为每个 Token 加入位置信息：

```text
Token Embedding：这个 Token 是什么
Positional Encoding：这个 Token 在哪里
```

位置编码可以帮助模型区分 Token 的前后顺序、相对距离和句子结构。词向量与位置编码组合后，模型才能同时利用语义和顺序信息。

### Decoder-Only 与 Encoder-Decoder 架构有什么区别？

原始 Transformer 使用完整的 Encoder-Decoder 架构，主要适用于机器翻译等输入到输出的转换任务：

```text
输入文本
   ↓
Encoder：理解完整输入
   ↓
上下文表示
   ↓
Decoder：根据 Encoder 的结果生成输出
```

Encoder 可以同时关注输入中前面和后面的所有 Token。Decoder 则包含：

- Masked Self-Attention：只能查看已经生成的内容，不能偷看未来 Token；
- Cross-Attention：读取 Encoder 的输出；
- Feed-Forward Network：进一步处理信息。

Decoder-Only 架构删除了独立的 Encoder 和 Cross-Attention，只保留并重复堆叠带有 Causal Mask 的 Decoder 风格模块：

```text
Prompt + 已经生成的内容
          ↓
Masked Self-Attention
          ↓
预测下一个 Token
```

用户输入和模型输出会被放在同一个 Token 序列中，模型不断根据已有内容预测下一个 Token，这叫作自回归生成（Autoregressive Generation）。

### 为什么很多主流生成式大语言模型采用 Decoder-Only？

1. **训练目标统一**：模型只需要学习“根据前面的 Token 预测下一个 Token”。问答、写作、翻译和代码生成都可以转换成文本续写问题。
2. **可以利用海量无标注文本**：互联网文本天然可以形成“已有文本 → 下一个 Token”的训练样本，不需要为每段文本人工标注。
3. **架构相对简单，容易扩展**：Decoder-Only 使用重复的同类模块，不需要维护独立的 Encoder 和 Cross-Attention，更适合扩大参数和训练规模。
4. **天然适合生成任务**：对话、文章和代码都需要逐 Token 生成，与 Decoder-Only 的自回归方式匹配。
5. **可以通过 Prompt 统一不同任务**：翻译、摘要、问答和代码检查都可以写成文字指令，放进同一个输入序列。

不过，不能绝对地说所有大语言模型都采用 Decoder-Only。更准确的说法是：许多主流的通用生成式大语言模型采用 Decoder-Only；Encoder-Only 仍适合分类和文本理解，Encoder-Decoder 仍适合翻译、摘要等输入到输出的转换任务。

### 小结

```text
Self-Attention：让 Token 动态获取上下文中的相关信息
Positional Encoding：补充 Token 的顺序和位置信息
Transformer：摆脱 RNN 的循环依赖，实现训练并行化
Encoder-Decoder：先理解输入，再生成输出
Decoder-Only：将生成任务统一为“根据前文预测下一个 Token”
```

Transformer 通过 Self-Attention 建立 Token 之间的上下文关系，通过位置编码补充顺序信息，并利用非循环结构实现训练并行化。Decoder-Only 又将多种生成任务统一成预测下一个 Token，因此成为许多通用生成式大语言模型的主流架构。

## 思考题 3：为什么不能只按“词”或“字符”分词？BPE 解决了什么问题？

模型不能直接处理文字，只能处理 Token ID。因此，我们需要先决定“什么算一个 Token”。

### 直接按词分词的问题

如果每个完整单词都是一个 Token，会出现两个主要问题：

- **词表爆炸**：语言中有大量单词、变形、专有名词和组合词。如果全部放进词表，Embedding 矩阵会非常大，训练和推理成本都会增加。
- **OOV（Out of Vocabulary）**：模型遇到词表中没有的新词时无法表示。例如，模型见过 `agent`，但未必见过 `agentic`、新公司名称或拼写变化。

### 直接按字符分词的问题

按字符分词几乎不会遇到 OOV，词表也比较小，但一句话会被切成很长的序列。序列越长，占用的上下文 Token 越多，Attention 计算、推理延迟和 API 成本也越高。单个字符承载的语义也比较弱，模型需要通过更多位置重新组合出词义。

### BPE 如何折中？

BPE（Byte Pair Encoding）的基本做法是从较小的基本单元开始，反复统计并合并语料中高频出现的相邻单元。例如：

```text
l o w
  ↓ 高频组合合并
lo w
  ↓
low
```

常见词可以合并成一个或少量 Token，不常见的新词仍可以拆成多个子词表示。因此 BPE 在以下两方面取得折中：

- 不需要为所有完整单词建立巨大词表，也能表示生僻词和新词；
- 序列通常比纯字符分词短，常见词和词根还能保留一定语义。

从 AI PM 角度，分词方式会影响上下文窗口、响应速度、API 成本和多语言效果。我们不需要手算 BPE，但要知道产品中的“字数”不等于 Token 数。

### 4.3 采样参数对比

三组参数使用同一条输入和 Zero-shot Prompt，每组重复 3 次。

| 采样策略 | JSON 成功率 | 字段准确率 | 必填字段完整率 | 幻觉字段 | 一致性 | 平均输入/输出 Token | 平均延迟 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| Greedy | 100% | 36.36% | 90.91% | 0 | 100% | 233 / 132.00 | 3.69 秒 |
| Balanced | 100% | 33.33% | 90.91% | 0 | 66.67% | 233 / 133.33 | 3.42 秒 |
| Creative | 100% | 33.33% | 96.97% | 1（3 次合计） | 33.33% | 233 / 136.33 | 3.54 秒 |

实际结果说明：

- 三组输出都能解析成 JSON，但“格式合法”不等于“字段正确”。
- Greedy 三次输出完全相同，最稳定，但字段准确率仍然不高。
- 温度升高后，输出一致性从 100% 下降到 33.33%，Creative 还出现了 1 个幻觉字段。
- 对结构化金融抽取这类确定性任务，更高的创造性没有产品价值，反而增加不稳定性。

### 4.4 Prompt 策略对比

三种 Prompt 都使用 Balanced 参数，每种重复 3 次。

| Prompt 策略 | JSON 成功率 | 字段准确率 | 必填字段完整率 | 幻觉字段 | 一致性 | 平均输入/输出 Token | 平均延迟 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| Zero-shot | 100% | 33.33% | 90.91% | 0 | 66.67% | 233 / 133.33 | 3.42 秒 |
| Few-shot | 100% | 100% | 100% | 0 | 100% | 415 / 124.00 | 4.02 秒 |
| 分步骤核对 | 100% | 33.33% | 96.97% | 0 | 66.67% | 259 / 146.67 | 4.08 秒 |

Zero-shot 的一个典型错误是：模型把 `1,200,000`、`450.20` 等数字输出成字符串，把币种输出成“港元”而不是 `HKD`，并漏掉值为 `null` 的 `cancellation_date`。这些输出看起来合理，但无法稳定进入下游数据库。

Few-shot 提供了一个相同 Schema 的正确示例后，三次实验都获得 100% 字段准确率和一致性。代价是平均输入从 233 Token 增加到 415 Token，延迟也略有上升。

分步骤核对 Prompt 没有明显改善这个 0.6B 小模型。这个结果提醒我：**Prompt 不是越长、越像“思维链”就越好，必须通过任务数据评测。** 对这个具体任务，展示目标格式和字段类型的示例，比抽象地要求模型认真检查更有效。

> 本实验样本只有一条，结论只用于理解实验方法，不能直接代表模型在真实金融公告数据集上的整体能力。正式上线前需要扩大到不同公司、事件类型、数字格式和缺失字段的测试集。

### 4.5 闭源模型与开源模型比较

| 维度 | 托管闭源模型 | 开源/开放权重模型 |
| --- | --- | --- |
| 性能 | 通常能快速获得较强的通用能力 | 取决于模型、硬件和微调，轻量模型复杂任务能力有限 |
| 成本 | 前期接入快，按 Token 持续付费 | 有硬件、部署、监控和维护成本，高调用量时可能更可控 |
| 可控性 | 只能使用厂商开放的参数和功能 | 可以私有部署、量化、微调和修改推理链路 |
| 隐私 | 数据需要发送给服务商，取决于合同和数据政策 | 可以让敏感数据留在企业网络内 |
| 运维 | 厂商负责扩缩容和模型升级 | 企业需要负责 GPU、性能、可用性、安全和升级 |
| License | 受 API 服务条款约束 | 不等于可以任意商用，必须逐个检查模型 License |

### 4.6 企业客服智能体如何选模型？

我会优先采用**混合方案**，而不是只押注一种模型：

- 托管闭源模型负责复杂问答、长尾表达理解和多轮对话；
- 小型私有模型负责意图分类、PII 脱敏、路由和稳定的结构化任务；
- 企业知识问题通过 RAG 连接经过审核的知识库；
- 退款、改地址、账户冻结等高风险操作接入确定性的 Workflow，并设置权限校验和人工审批。

最终选择需要通过真实客服数据比较正确率、拒答率、延迟、单次成本、峰值并发、隐私合规、可观测性和故障降级能力。模型能力只是系统的一部分。

## 思考题 5：如何缓解模型幻觉？

### 5.1 重点方案：RAG

RAG（Retrieval-Augmented Generation，检索增强生成）的工作过程是：

```text
用户问题
  ↓
从可信知识库检索相关内容
  ↓
把问题和证据一起放进上下文
  ↓
要求模型只根据证据回答并返回引用
```

它适合企业知识库问答、客服、论文阅读和政策查询。它的主要价值是把答案依据从模型模糊的“参数记忆”转移到可更新、可追溯的外部资料。

但 RAG 不能自动消灭幻觉。如果文档解析错误、切分不合理、检索不到正确资料，或者模型没有忠于证据，仍然会答错。因此需要同时评估 **Retrieval Quality** 和 **Answer Faithfulness**。

### 5.2 其他改进方法

| 方法 | 机制 | 优势 | 限制 | 适用场景 |
| --- | --- | --- | --- | --- |
| [Self-RAG](https://arxiv.org/abs/2310.11511) | 训练模型使用特殊反思 Token，自主判断是否需要检索，并评价证据和回答 | 不必每次固定检索，可提升检索使用的灵活性和引用质量 | 需要专门训练，工程和数据成本高 | 开放域问答、长文本生成 |
| [Corrective RAG](https://arxiv.org/abs/2401.15884) | 先用检索评估器判断文档质量；证据不够时触发纠正、过滤或外部搜索 | 能处理“检索结果本身不可靠”的问题 | 增加评估器、搜索成本和延迟，外部搜索也可能不可信 | 知识变化快、企业库覆盖不足的问答 |
| [Chain-of-Verification](https://aclanthology.org/2024.findings-acl.212/) | 先生成草稿，再规划验证问题，独立回答验证问题，最后修订答案 | 不改模型也能在推理阶段加入事实核验 | 多次调用增加 Token 和延迟；模型可能重复自己的错误 | 高价值长文本、事实密集型回答 |
| [F-DPO：Factuality-Aware Preference Learning](https://aclanthology.org/2026.findings-acl.1968/) | 在训练阶段给偏好样本加入事实性标签，纠正事实顺序错误的偏好对，并强化事实差异 | 从训练阶段降低“流畅但错误”的偏好，不需要额外 Reward Model | 需要可靠的事实标签和训练资源，论文结果不能直接保证所有业务数据有效 | 有能力训练或微调基座模型的团队 |

2026 年的 F-DPO 代表了另一条路线：RAG、工具调用和验证主要在推理/系统阶段减少幻觉，而 F-DPO 尝试在训练阶段让模型更偏好真实答案。论文报告它在多个开放权重模型上降低了幻觉，但产品上线仍需要在自己的业务数据上重新验证。

作为 AI PM，我会把“降低幻觉”拆成四层：

1. **知识层**：知识源是否可信、及时、版本明确；
2. **检索层**：是否找到了正确且完整的证据；
3. **生成层**：回答是否忠于证据，证据不足时是否拒答；
4. **验证层**：引用能否支持结论，高风险答案是否需要规则或人工审核。

## 思考题 6：论文辅助阅读智能体设计

### 6.1 基座模型选择

MVP 阶段我会把当前的 **Gemini 3.1 Pro Preview** 作为首个候选模型，而不是未经评测就确定为最终生产模型。根据 [Gemini 3.1 Pro 官方模型页](https://ai.google.dev/gemini-api/docs/models/gemini-3.1-pro-preview)，它支持 PDF 输入、约 1M Token 输入上下文、Structured Outputs 和 Function Calling；[Document Understanding 文档](https://ai.google.dev/gemini-api/docs/document-processing) 也说明 Gemini 可以理解 PDF 中的文本、图表和表格。

这些能力适合论文阅读，但选择时仍需要考虑：

- 论文问答、摘要和跨论文比较的真实准确率；
- PDF、公式、表格、图片和 OCR 的解析能力；
- 上下文长度不等于有效理解长度，需要测试长文中部信息是否会丢失；
- API 成本、延迟、并发、数据保留政策和地区可用性；
- Preview 模型的版本稳定性、迁移成本和 Vendor Lock-in；
- 工具调用、结构化输出、拒答和引用能力。

上线前应建立真实论文评测集，同时比较至少一个闭源模型和一个可私有部署的开放权重模型。

### 6.2 系统架构

```text
PDF 上传
  ↓
文本解析 / OCR / 版面与表格识别
  ↓
按论文、章节和段落切分，保存页码与章节元数据
  ↓
Embedding + 向量索引
  ↓
关键词与向量混合检索
  ↓
Reranker 重新排序证据
  ↓
LLM 基于证据回答、总结或比较
```

长论文不应该简单粗暴地全部塞进 Prompt。我会使用：

- **章节级 RAG**：先定位相关章节，再检索段落；
- **分层摘要**：先生成段落摘要，再合成章节摘要和全篇摘要，同时保留原文链接；
- **Query Decomposition**：把复杂问题拆成多个可检索的小问题；
- **Evidence Card**：多论文比较时，先为每篇论文建立包含研究问题、方法、数据、结论、限制和证据位置的卡片，再进行跨论文综合。

### 6.3 Prompt 设计

核心 System Prompt 可以包含：

```text
你是严谨的论文阅读助手，只能依据提供的论文证据回答。
区分“作者原文结论”“你的归纳”和“无法从论文确认的信息”。
每个事实性结论必须返回论文标题、页码、章节和证据片段。
如果证据不足或多篇论文存在冲突，明确说明，不得补全或猜测。
按指定 JSON Schema 输出，方便系统校验引用。
```

对于不同任务再增加具体 User Prompt：

- 摘要：分别抽取研究问题、方法、数据、核心结果、贡献和限制；
- 问答：先检索证据，再回答，并标注无法确认的部分；
- 信息抽取：定义字段、类型、缺失值和示例；
- 多论文比较：统一比较维度，先逐篇取证，再综合相同点与冲突点。

### 6.4 如何保证准确、客观、忠于原文？

系统中需要加入以下设计：

1. **细粒度引用**：每个事实结论绑定论文、页码、章节、段落 ID 和短证据片段；
2. **引用校验**：检查引用是否真实存在，以及证据是否能支持对应结论；
3. **无法回答时拒答**：检索分数或证据覆盖不足时返回“原文无法确认”；
4. **独立验证**：用第二次模型调用或不同模型逐条核对结论与证据，避免只让原模型自我确认；
5. **来源隔离**：清楚区分论文原文、引用文献、外部搜索结果和模型推断；
6. **人工审核**：文献综述、医疗结论、投资判断等高风险输出必须由研究人员确认；
7. **审计日志**：记录模型版本、Prompt、检索证据、参数和最终输出，保证可追溯。

### 6.5 验收指标

| 指标 | 要回答的问题 |
| --- | --- |
| Citation Precision | 返回的引用中，有多少真的支持对应结论？ |
| Faithfulness | 回答中的事实是否都能从提供的论文证据中推出？ |
| Answer Correctness | 答案与人工标准答案有多一致？ |
| Coverage | 论文的关键问题、方法、结果和限制是否覆盖完整？ |
| Abstention Accuracy | 证据不足时能否正确拒答，而不是编造？ |
| Latency | 单篇摘要、单次问答和多论文比较需要多久？ |
| Cost per Task | 每种任务平均消耗多少 Token 和 API 成本？ |

作为 AI PM，我不会把验收标准写成“回答看起来不错”。应该先建立带证据和标准答案的论文测试集，再为不同任务设定准确率、引用质量、延迟和成本阈值。
